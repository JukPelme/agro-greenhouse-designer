# Архитектурные решения

Короткие заметки «почему так, а не иначе» — для собеседования.

## 1. Почему LangGraph, а не CrewAI / AutoGen

CrewAI и AutoGen хороши, когда нужно «агенты болтают между собой и приходят к
решению». Здесь задача другая — детерминированный pipeline с явным state и
условными переходами (Validator решает, нужна ли ещё итерация Designer).
LangGraph даёт прямой контроль над графом и явный typed state — это и
показывается в коде.

## 2. Почему Pydantic-state, а не TypedDict

Pydantic-state валидируется на каждом шаге. Если Designer вернул блок с
негативной высотой — упадёт прямо в Designer, а не через три нод. Это
снижает время отладки на порядок и заодно даёт `model_dump_json` для
демо-кэша из коробки.

## 3. Почему LLM не считает числа

Эмпирически: даже Claude Opus с явной формулой даёт расхождение в 10–30%
на тривиальной арифметике. Для проектных расчётов это категорически
неприемлемо. Поэтому Engineer — чистый Python с pytest на каждую функцию,
а LLM только выбирает *какую* функцию вызвать через структурированный
вывод Designer-а.

## 4. Почему rules.yaml, а не «попросим LLM проверить СП»

Тот же аргумент: LLM не должен интерпретировать норматив. Машинопроверяемые
правила выписаны вручную с явной ссылкой на пункт СП. RAG нужен только
для одного — достать дословный текст пункта и приложить к issue. Это
основа доверия к Validator.

## 5. Почему демо-режим из pickle, а не из LangGraph checkpoint

LangGraph checkpoints — для возобновления прерванного выполнения внутри
сессии. Демо нужен «один сохранённый прогон → проигрывание». Pickle
проще, не требует postgres/sqlite, помещается в репо.

## 6. Что я бы изменил для production

- ChromaDB → Qdrant или pgvector с миграциями
- pickle для кэша → S3 + версионирование
- Streamlit → FastAPI + React (для multi-user)
- rules.yaml → DSL на Lark с авто-извлечением части правил из PDF
- Расчёты → разделить «pre-design» и «detailed» с интеграцией EnergyPlus
